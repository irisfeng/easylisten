# TTS 选型评估:VoxCPM 及备选方案

面向"轻听"的朗读需求,评估把 [OpenBMB/VoxCPM](https://github.com/OpenBMB/VoxCPM) 作为
高质量神经 TTS 引擎的可行性,并给出落地路线。

## 需求

- **中文为主、中英混排**:文章含专有名词、数字、中英夹杂。
- **长文朗读**:一篇数百到数千字,需要分句流式合成,边合成边播放。
- **句级进度**:播放器要高亮"正在读的句子",引擎须按句回调。
- **成本可控**:先零成本跑通体验,再按需接入高质量音色。

## VoxCPM 能力

| 维度 | 情况 |
| --- | --- |
| 规模 | 20 亿参数,MiniCPM-4 基座,tokenizer-free 扩散自回归架构 |
| 语言 | 30 种语言(含中、英)+ 9 种中文方言 |
| 音质 | 48kHz 录音棚级输出 |
| 能力 | 文字描述生成音色(voice design)、短样本音色克隆、带风格控制的"极致克隆" |
| 流式 | 支持;RTX 4090 上 RTF ≈ 0.3(nano-vLLM 可到 ~0.13) |
| 显存 | 标准推理约 8GB VRAM,需 CUDA 12.0+ |
| 部署 | vLLM-Omni 提供 **OpenAI 兼容 `/v1/audio/speech`**;另有 nano-vLLM、llama.cpp-omni(CPU/Metal/CUDA/Vulkan) |
| 许可 | Apache-2.0,可商用 |

### 结论:**技术上完全可行,但需自建 GPU 服务**

- **优点**:开源可商用、48kHz 音质、支持中文与音色克隆、原生流式、且提供 OpenAI 兼容
  接口——意味着可以用极小的适配层接入,后续换别的兼容后端也不必改前端。
- **代价**:需要一块 ≥8GB 显存的 GPU 常驻,或用 llama.cpp-omni 在 CPU 上跑(延迟更高)。
  对纯静态托管(如 Vercel)的当前形态,需要额外的推理服务(自建或 GPU 云函数)。

## 落地路线(分阶段)

引擎已在代码中抽象为可插拔接口 `SpeechEngine`(见 `src/lib/tts.ts`)。播放器只依赖这个
接口,不关心背后是谁在发声。

**阶段一(已落地)——浏览器 Web Speech API**
零成本、零延迟、离线可用,先把"选文 → 分句朗读 → 句级高亮 → 变速/跳句"的完整体验跑通。
局限:音色取决于用户系统自带的语音包,质量参差。

**阶段二——接入 VoxCPM(推荐)**
新增一个 `VoxCPMEngine implements SpeechEngine`,把句子逐句 POST 到自建的
`/v1/audio/speech`(vLLM-Omni 起服务),用隐藏的 `<audio>` 播放返回音频,在每段音频的
`play`/`ended` 事件里触发和现在完全相同的 `onSentence`/`onDone` 回调。**播放器与页面代码
无需改动**。可预取下一句音频以掩盖合成延迟。

**阶段三——音色与个性化**
利用 VoxCPM 的 voice design / 克隆能力,为不同频道(科学、新闻、深度、文化)配不同音色,
或允许用户上传样本克隆自己的声音。

## 备选方案对照

| 方案 | 音质 | 中文 | 成本 | 备注 |
| --- | --- | --- | --- | --- |
| **Web Speech API**(当前) | 中 | 依赖系统 | 免费 | 无需后端,先跑通体验 |
| **VoxCPM**(推荐目标) | 高(48kHz) | 强 | 需 GPU | 开源可商用、OpenAI 兼容、可克隆 |
| 云 TTS(如各家商用 API) | 高 | 强 | 按量付费 | 无需运维,但有单价与数据外发 |

**建议**:保留当前 Web Speech 作为零成本兜底,把 VoxCPM 作为主推的高质量引擎,通过
`SpeechEngine` 接口在两者间平滑切换。
